1. Resumen ejecutivo

MiKiwi es una aplicacion web de comercio electronico con fuerte componente de personalizacion visual. El sistema se construye sobre Laravel 12 y React 18, enlazados mediante Inertia.js, lo que permite una experiencia SPA-like sin abandonar el control de rutas y middleware en servidor.

La arquitectura observada ofrece una base profesional: separacion por capas, repositorios por contrato, servicios de negocio, validaciones formales en flujos relevantes, autorizacion con policies y soporte de asincronia por eventos/listeners/jobs.

Tambien se detectan riesgos que requieren accion inmediata: exposicion de secretos en documentacion, inconsistencias de implementacion en checkout/direcciones, contratos JSON no estandarizados y ausencia de automatizacion CI declarativa en el repositorio.

2. Introduccion y contexto del proyecto

2.1 Contexto funcional

El producto MiKiwi persigue habilitar un flujo completo de venta digital para articulos personalizables. A nivel funcional integra catalogo, carrito, checkout, area de usuario y panel de gestion, anadiendo un configurador visual que impacta directamente en conversion y diferenciacion comercial.

2.2 Contexto tecnologico

  • Backend: Laravel 12, PHP 8.2+, Sanctum, Eloquent ORM.
  • Frontend: React 18, Inertia.js 2, Tailwind CSS 3, Vite 7.
  • Integraciones: Stripe (pagos), Cloudinary (media), Correos (pickup).
  • Asincronia: queues, jobs, events/listeners y comandos programables.

2.3 Proposito de la memoria

La memoria cumple tres objetivos simultaneos: (1) soporte academico/profesional de defensa, (2) referencia tecnica para mantenimiento y roadmap, (3) base de trazabilidad para decisiones de arquitectura y priorizacion de deuda tecnica.

2.4 Alcance funcional detallado del producto

  • Canal de venta con navegacion por categorias y detalle de producto.
  • Carrito persistente por sesion con validaciones de stock.
  • Checkout con seleccion de direccion y flujo de pago externo.
  • Gestion de cuenta: perfil, direcciones, historial de pedidos.
  • Backoffice para contenido visual, catalogo y operaciones administrativas.
  • Configurador visual de producto con composicion por partes y capas.

2.5 Premisas arquitectonicas

  • Evitar duplicidad de logica entre frontend y backend en validacion de negocio.
  • Mantener el backend como fuente de verdad de permisos y estado transaccional.
  • Permitir evolucion incremental hacia mayor desacople sin reescritura total.
  • Priorizar correccion y trazabilidad frente a complejidad prematura.

5. Arquitectura del sistema

5.1 Patron de arquitectura

Se adopta un monolito modular con aproximacion hibrida web/API. Inertia funciona como puente entre backend y frontend para navegacion principal, mientras JSON API cubre casos asincronos puntuales.

5.2 Capas y responsabilidades

Capa Implementacion principal Responsabilidad
Entrada HTTP Routes + Middleware + Controllers Recepcion de peticiones, seguridad perimetral y orquestacion
Aplicacion Services + Actions Reglas de negocio y coordinacion de casos de uso
Datos Repositories + Models Eloquent Persistencia, consultas y relaciones de dominio
Integracion StripeService, CloudinaryService, CorreosService Interaccion con proveedores externos
Gobierno Policies, FormRequests, Exceptions Autorizacion, validacion y consistencia de errores

5.3 Flujo de peticion de referencia

  1. Cliente inicia accion por Inertia o Axios.
  2. Laravel resuelve ruta y ejecuta middleware.
  3. Controller valida/autoriza y delega a servicio o accion.
  4. Servicio consume repositorio/modelo o servicio externo.
  5. Respuesta se serializa como Inertia page payload o JSON API.

5.4 Decisiones y trade-offs

  • Inertia + React: reduce complejidad full-stack, pero aumenta acoplamiento de releases backend/frontend.
  • Service + Repository: mejora mantenibilidad, pero su aplicacion parcial introduce deuda de coherencia.
  • Eventos/Jobs: habilita crecimiento asincrono, aunque parte de handlers estan en estado base.
  • API selectiva: aporta flexibilidad, con coste de estandarizar contratos entre canales.

5.5 Vista C4 resumida (documental)

Nivel Elemento Responsabilidad Observacion tecnica
Contexto MiKiwi Platform Venta y personalizacion de productos Sistema central del negocio
Contenedor Laravel App Dominio, seguridad, persistencia e integraciones Monolito modular
Contenedor React UI (Inertia pages) Interfaz de usuario y flujo interactivo Server-driven navigation + llamadas axios puntuales
Contenedor externo Stripe / Cloudinary / Correos Pagos, media y puntos de recogida Integracion por servicios dedicados

5.6 Flujo tecnico de checkout (pasos de referencia)

  1. Usuario confirma carrito y direccion de envio.
  2. Frontend invoca endpoint de preparacion de pago.
  3. Backend valida datos, stock y contexto de usuario.
  4. Se crea PaymentIntent y se retorna secreto de cliente.
  5. Frontend confirma pago con SDK Stripe.
  6. Backend persiste pedido, items, snapshots y estados.
  7. Eventos/listeners disparan tareas complementarias (notificacion, inventario).

6. Entorno servidor (Laravel)

6.1 Base framework

El servidor opera con Laravel 12 y PHP 8.2+, con composicion modular en app/. Se observan middleware y providers orientados a auth, Inertia y resolucion de dependencias por interfaces.

6.2 Rutas y exposicion

Canal web

routes/web.php concentra navegacion de sitio, checkout, perfil, paneles administrativos y paginas de soporte legal. Se identifican tambien endpoints con prefijo /api/* definidos en este canal para necesidades operativas concretas.

Canal API

routes/api.php contiene endpoints de productos y autenticacion API con throttle para operaciones sensibles.

6.3 Capa de dominio y persistencia

Los modelos principales de ecommerce incluyen User, Product, Category, Order, OrderItem, UserAddress, Coupon y PickupPoint. El esquema se evoluciona por migraciones versionadas, con presencia de UUID y campos JSON para snapshots donde aplica.

6.4 Seguridad, validacion y autorizacion

  • Autenticacion web por session guard y middleware auth.
  • Autenticacion API por Sanctum en rutas protegidas.
  • Policies de usuario, pedidos y direcciones.
  • FormRequests presentes en varios flujos, con mezcla de validacion inline en otros puntos.

6.5 Asincronia y procesamiento diferido

La plataforma incorpora eventos, listeners y jobs en cola, junto con comandos scheduler para tareas operativas (stock, limpieza, sincronizacion). Este pilar esta implementado como base de escalabilidad, aunque ciertos handlers requieren completar logica de negocio.

6.6 Observabilidad

Existe logging funcional en capas de aplicacion e integracion. No se detecta instrumentacion APM o tracing distribuido en el estado actual del repositorio.

6.7 Matriz de componentes backend por criticidad

Componente Criticidad Rol Estado
OrderController / OrderService Alta Checkout y creacion de pedido Requiere estabilizacion prioritaria
CartController / CartService Alta Preparacion de compra y validacion de stock Con hallazgos de consistencia
UserAddressController / Service Alta Gestion de direcciones de envio Con errores de variable detectados
ProductService / Repositories Media Catalogo y consulta de productos Base estable
CloudinaryService Media Pipeline de media y optimizacion visual Funcional con complejidad alta

6.8 Gobierno de errores en servidor (objetivo recomendado)

Para elevar robustez, se recomienda un envelope uniforme de respuesta de error en API: { success: false, code, message, details, trace_id }. En el estado actual, la estructura varia por endpoint y controlador.

7. Entorno cliente (React + Inertia)

7.1 Inicializacion y estructura

La app React se inicializa con createInertiaApp en resources/js/app.jsx. El bootstrap de HTTP se define en resources/js/bootstrap.js con Axios global. Se observa uso de context para idioma y carga perezosa de componentes de soporte.

7.2 Comunicacion asincrona

  • Inertia para navegacion y formularios server-driven.
  • Axios para llamadas JSON en flujos puntuales (checkout, gestion de recursos, etc.).
  • Ziggy para helper de rutas y consistencia de endpoints.

7.3 Estado y patrones de composicion

Se prioriza estado local por hooks. Se verifica LanguageContext y hook custom usePartOptimization. No se identifica una capa formal de servicios frontend ni cliente Axios central avanzado.

7.4 UI, rendimiento y experiencia

  • Tailwind CSS como base de estilo y responsive.
  • Vite 7 para build y dev server de alto rendimiento.
  • Integracion de React Three Fiber para funcionalidades visuales avanzadas.
  • Lazy loading en componentes de coste elevado.

7.5 Limitaciones detectadas

  • No existe carpeta resources/js/Services.
  • No existe resources/js/Utils/axios.js como capa central.
  • El manejo de errores HTTP puede variar segun pagina/componente.

7.6 Arquitectura de front recomendada para siguiente iteracion

Bloque Estado actual Mejora recomendada
Servicios HTTP Llamadas axios repartidas Crear capa services por dominio y cliente central
Manejo de errores No uniforme Interceptor + normalizador de errores UI
Estado compartido Principalmente local Consolidar estado transversal minimo (auth/cart/ui)
Observabilidad cliente Basica Instrumentar eventos UX y errores JS

7.7 Consideraciones de rendimiento frontend

  • Definir presupuesto de bundle por pagina critica.
  • Diferir librerias pesadas solo en rutas que las necesiten.
  • Medir TTI y CLS en flujo real de checkout y configurador.
  • Revisar politicas de cache para assets y media transformada.

8. Modelo de datos y persistencia

8.1 Estrategia de modelado

El modelo persiste entidades de negocio en una base relacional, evolucionada por migraciones incrementales. Se observan claves UUID en gran parte del dominio y cast de tipos para consistencia de lectura/escritura.

8.2 Entidades y relaciones clave

  • User 1..N Order
  • Order 1..N OrderItem
  • User 1..N UserAddress
  • Category 1..N Product
  • Product N..N Product (accesorios)

8.3 Observaciones de consistencia

El uso de snapshots JSON para direccion de envio/facturacion es adecuado para trazabilidad historica de pedidos. No obstante, se detectan potenciales divergencias entre algunos payloads de checkout y campos persistidos, lo que requiere normalizacion de contrato.

8.4 Reglas transaccionales criticas

  • Creacion de pedido e items dentro de transaccion atomica.
  • Validacion de stock previa al decremento de inventario.
  • Persistencia de snapshots para preservar evidencia historica del pedido.
  • Actualizacion de estado de pago desacoplada para soportar reintentos.

8.5 Riesgos de integridad de datos

Riesgo Causa tecnica Impacto Mitigacion
Divergencia de campos de pedido Payload no alineado con modelo/migracion Datos omitidos o inconsistentes Contrato unico + pruebas de integracion
Stock desincronizado Errores en flujo transaccional Sobreventa o anulaciones Bloqueo transaccional y pruebas concurrentes
Historico incompleto Manejo parcial de estados de pago/envio Baja trazabilidad operativa Eventos de estado + auditoria de cambios

9. Integraciones externas

9.1 Stripe

Integracion backend mediante SDK oficial y componentes frontend para la captura de pago. La arquitectura es correcta, pero el flujo de endpoint de payment intent requiere cierre de inconsistencias en controlador/ruta.

9.2 Cloudinary

Integracion mediante SDK oficial para gestion de activos, consulta y transformacion de imagenes. Es una decision acertada para rendimiento y simplificacion de media pipeline.

9.3 Correos

Servicio de puntos de recogida con fallback mock, adecuado para continuidad funcional en entorno de desarrollo y resiliencia ante indisponibilidad externa.

9.4 Matriz de integraciones y resiliencia

Proveedor Tipo Operacion critica Riesgo Estrategia recomendada
Stripe Pago Creacion/confirmacion de intento de pago Fallo en cierre de transaccion Idempotencia + reconciliacion por webhook
Cloudinary Media Carga y transformacion de assets Latencia o error de recurso Fallback de imagen + reintento controlado
Correos Logistica Consulta pickup points No disponibilidad puntual Cache local y fallback mock temporal

10. Seguridad y cumplimiento tecnico

10.1 Controles implementados

  • Middleware de autenticacion y rol admin.
  • Policies para autorizacion por ownership/rol.
  • FormRequests en flujos relevantes.
  • Sanctum en canal API protegido.
  • Protecciones nativas de Laravel frente a CSRF y SQL injection por ORM.

10.2 Brechas y riesgos de cumplimiento

  • Secretos en documentacion versionada.
  • Ausencia de politica formal de secretos en repositorio.
  • Inconsistencia de validacion segun endpoint.
  • Falta de estandar unificado de errores JSON y trazabilidad request-level.
Recomendacion inmediata: retirar y rotar credenciales expuestas antes de cualquier despliegue o demostracion publica.

11. Calidad, pruebas y validacion

11.1 Estado de pruebas automatizadas

La suite de PHPUnit esta presente con separacion Feature/Unit. Se observa cobertura funcional en servicios, repositorios, requests, policies, jobs y eventos. Este punto representa una fortaleza estructural del proyecto.

11.2 Procesos de calidad actuales

  • Ejecucion local de pruebas y build.
  • Revision manual de comportamiento en entorno de desarrollo.
  • Sin pipeline CI declarativa en repositorio para enforcement continuo.

11.3 Limitaciones

  • Sin baseline automatizado de rendimiento/accesibilidad.
  • Sin consolidacion de quality gates por pull request.
  • Necesidad de reforzar pruebas end-to-end en checkout critico.

11.4 Estrategia de pruebas recomendada por capas

Capa Tipo de prueba principal Objetivo
Dominio (Services/Actions) Unit + feature orientada a caso de uso Validar reglas de negocio e invariantes
HTTP/API Feature de contrato y autorizacion Garantizar codigos, payload y permisos
Persistencia Feature con factories/migraciones Consistencia de modelos, relaciones y cast
Flujo end-to-end critico E2E (pendiente de formalizacion) Asegurar checkout completo sin regresion

11.5 Criterios de salida de release (propuestos)

  • Sin hallazgos criticos abiertos de seguridad o estabilidad.
  • Pruebas de auth, carrito y checkout en verde.
  • Validacion manual de rutas administrativas sensibles.
  • Checklist de secretos y variables de entorno completado.

12. Operacion, despliegue y mantenimiento

12.1 Operacion local

  1. Instalacion de dependencias PHP y JS.
  2. Configuracion de variables de entorno.
  3. Migraciones y seeders de base de datos.
  4. Arranque por composer dev o npm start.

12.2 Mantenibilidad operativa

La base de comandos es funcional y permite desarrollo concurrente, pero no se ha formalizado aun un playbook completo de incidentes, rollback y verificacion de release.

12.3 Condiciones para production-ready

  • Resolucion de hallazgos criticos de seguridad y estabilidad.
  • Estandar de contratos API y manejo de errores.
  • CI con pruebas y build en cada integracion.
  • Evidencia objetiva de performance/accesibilidad.

12.4 Runbook operativo minimo (propuesto)

  1. Validar estado de base de datos y cola antes de despliegue.
  2. Aplicar migraciones con ventana controlada y backup previo.
  3. Publicar nueva version y verificar health checks funcionales.
  4. Ejecutar smoke test en login, catalogo, carrito y checkout.
  5. Monitorear logs de error durante 30-60 minutos post-release.

12.5 Protocolo de incidente (propuesto)

Fase Tiempo objetivo Accion
Deteccion < 5 min Identificar impacto, modulo y version afectada
Contencion < 15 min Mitigar flujo critico (feature flag, rollback, desvio)
Resolucion < 2 h Hotfix validado con pruebas dirigidas
Postmortem < 48 h Documento RCA + accion preventiva trazable

14. Riesgos arquitectonicos y deuda tecnica

14.1 Riesgos principales

  1. Deriva de arquitectura por coexistencia de patrones y excepciones no normalizadas.
  2. Riesgo operativo en checkout por incoherencias de implementacion.
  3. Riesgo de seguridad documental con exposicion de secretos.
  4. Riesgo de integracion por contratos API no estandarizados.
  5. Riesgo de calidad por ausencia de CI y quality gates automatizados.

14.2 Deuda tecnica prioritaria

  • Correccion estructural de controladores y servicios criticos.
  • Unificacion de validaciones en FormRequests.
  • Estandar de respuesta JSON (success/error/codigo/trazabilidad).
  • Centralizacion de cliente HTTP frontend e interceptores.
  • Politica de secretos con escaneo y rotacion periodica.

  1. Workstream A - Estabilidad transaccional: checkout, carrito, direcciones y auth API.
  2. Workstream B - Seguridad y gobierno: secretos, hardening y estandar documental.
  3. Workstream C - Calidad de entrega: CI, pruebas de contrato, baseline de performance.
  4. Workstream D - Plataforma frontend: capa services, manejo de errores y telemetria basica.

15.2 Gobernanza de ejecucion del plan

  • Revision semanal de riesgos abiertos con responsables asignados.
  • Tablero unico de deuda tecnica con fecha objetivo y evidencia de cierre.
  • Criterio de DoD explicito por modulo critico.
  • Reporte quincenal de KPIs de calidad y seguridad al equipo completo.

16. Conclusiones

MiKiwi presenta una base arquitectonica valida para evolucionar a estandares de produccion con un esfuerzo controlado. El stack es adecuado y la estructura modular ya existe en terminos de capas, componentes y pruebas.

El reto principal no es de cambio de tecnologia, sino de consolidacion tecnica: resolver inconsistencias en flujos criticos, normalizar contratos, reforzar seguridad documental y automatizar calidad de forma continua.

Si se ejecuta el plan 30/60/90 dias, el proyecto puede pasar de estado funcional con deuda a una plataforma robusta, trazable y defendible en auditoria tecnica formal.

Desde una perspectiva de arquitectura empresarial, la recomendacion es evolucion incremental con foco en confiabilidad operacional. Es decir: primero asegurar integridad de compra y seguridad, despues consolidar estandares de interfaz y automatizacion de calidad, y finalmente optimizar observabilidad y eficiencia de entrega.

Con ese orden de prioridades, el proyecto puede mejorar significativamente sin incurrir en reescrituras costosas ni degradar la capacidad de entrega funcional del equipo.
